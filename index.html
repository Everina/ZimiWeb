问题

Agent 能"对话"，
却无法真正"执行"

Agent 生态里有一个普遍的断点：大量 Agent 已经具备对话能力，却无法完成实际操作。根源是两个卡点。

没有可信的 Skill

有想法，但没有经过认证的工具，只能输出文字建议，无法触达真实系统。

没有真实身份资产

无法绑定邮箱 / 手机号 / API 钱包，自动化链路在最后一公里断掉。

这个问题是在嗅嗅豹的实践中浮现的：我们把"商品营销视频生成"封装为可复用 Skill，随即发现这种能力单元在开发者生态中广泛存在，但没有标准化的分发与协作市场。

发现

一条完整的失败链路

对用户行为的拆解，归纳出从意图到放弃的完整失败链路——每一个环节都是一个用户流失点。

核心判断：Skill 必须从"可搜索的工具集合"升级为"可推荐、可验证、可运行、可进化的能力矩阵"。仅有能力还不够——Agent 必须同时具备对 Skill 的配置能力与可信的身份资产，才能在复杂场景中完整执行。

决策

针对每个环节，
做对应的产品设计

不会找 SkillFinder 语义检索

从关键词检索升级为基于任务理解的推荐（intent → skill），结合 Agent 真实运行数据动态排序。

嗅嗅豹

面向跨境时尚非标品选品场景的 AI 分析平台。两期都围绕同一类选品问题展开：一期服务内部 Lazada 和 1688 行业运营，验证多 Agent 能否完成商机分析；二期面向外部跨境商家，把高频分析需求沉淀为稳定工作流。

核心洞察
  • 商机定义取决于商家类型，产品要先判断谁在选品，再定义什么算机会。
  • 数据处理链路和搜索策略强耦合，类目映射、CPV 拍平和实体清洗会直接影响召回与排序质量。
  • 高频需求应该工作流化，Agent 处理长尾，真实业务更需要稳定、可评测、低成本的分析路径。
  • Agent 产品需要数据飞轮，把用户反馈和经营结果回收为需求理解、召回、排序和分析能力的优化信号。
一期抽象

内部真实选品需求，本质是多平台数据对齐和商品特征工程。

我们发现行业运营不是在“找一个商品”，而是在做一套判断：把 Shein、Temu、Shopee most popular 和 new arrival 榜单的商品收集起来，再抽取卖得多、增长快的商品特征趋势。这个过程非常适合用 Agent 框架拆成 Planning、Data、Analysis 三层。

Planning
把用户模糊需求结构化

我主导 Prompt 设计，将自然语言需求拆成类目、平台、趋势方向等结构化维度，并用运营评分做校验。

Data
两路数据源：离线全量 + 在线榜单

离线使用 Shopee 类目全量数据做 embedding 和 recall 评测；在线接入实时榜单，类目映射是关键难点。

Analysis
基于结构化维度做特征提取和排序

围绕工具调用、特征提取和排序结果做评估，重点看 Agent 是否正确调用工具并产出可解释结果。

一期问题

一期获得 1688 大力支持，但也暴露出三个系统性问题

一期证明了 Agent 能跑通商机分析链路，但上线后也发现：链路过长导致成本失控，能力承诺超过实际可执行逻辑，报告缺少推导过程导致用户信任不足。这些问题推动了后续产品方向的转折。

链路过长 → 成本失控

在线获取当天商品数据平均耗时 42 分钟，多环节依赖导致成本、等待时间和失败风险同时上升。

能力承诺 > 实际实现

结构化分析在体验上被用户理解为“会做完整决策”，但实际很多分析还没有真正沉到可执行逻辑里。

信任缺失

报告给出了结论，但推导过程不够透明，用户无法判断哪些数据、特征和排序逻辑支撑了最终判断。

架构转向

从“通用多 Agent”，转向“Workflow + Agent 兜底”

用户表面表达很多样，但底层高频分析模式可以沉淀为电商、社媒、品牌三类数据源，以及大约 12 类固定工作流。高频需求工作流化，长尾需求再交给 Agent。

二期落地

真正影响商家选品效果的，
不是 Agent 本身，而是数据源质量

外部商家的选品会同时参考平台榜单、社媒趋势、广告投放、达人内容和搜索热度。所以二期重点从“继续增强 Agent”转向“多源数据处理能力”：把不同来源的数据接入、清洗、映射，并拍平到统一 CPV 集上，再把搜索效果和数据质量做耦合优化。

数据接入 多源数据接入与清洗

接入平台榜单、社媒趋势、达人内容、品牌数据和搜索热度，先做字段清洗、去重和实体标准化。
类目映射 多级类目映射表 + LLM 生成评估闭环

不同平台类目体系差异很大，类目映射决定了数据能否比较。我设计多级类目映射表，并用 LLM 生成候选映射，再通过评估闭环校验。
CPV 拍平 多元异构数据统一到一套 CPV 集

将商品、帖子、达人和品牌数据里的颜色、材质、风格、场景等特征统一沉淀，降低后续召回和排序的语义噪声。
搜索优化 搜索召回、排序优化与质量评测耦合

数据质量和搜索策略相互影响，因此我们围绕类目准确率、相关性准确率和召回质量做多轮评测和迭代。
输出形态

二期产出：三类数据源、固定工作流和可量化搜索质量

二期把高频选品需求沉淀为三类报告入口：电商趋势、社媒趋势和品牌趋势。与一期相比，重点不再是让 Agent 自由规划，而是让数据源、搜索召回和排序分析都能被稳定评测。

内测反思

内测之后，我们重新判断下一步机会在哪里

01
为什么做商品视频生成

选品分析相对低频，商家更高频的需求是找到同款爆款视频，并复刻成自己的商品营销内容。

02
虾小宝的能力溢出

商品视频生成被封装成 Skill 后，说明能力可以从单点工具沉淀为可复用的 Agent 能力单元。

03
中文 Skill 社区

市面上缺少成熟的中文 Skill 社区，因此我们开始思考如何做能力分发、验证和协作。

04
身份配置继续补齐

Agent 不只需要 Skill，还需要身份资产、账号和外部系统配置，才能在真实世界完整执行任务。

商品视频生成

从低频选品分析，延展到更高频的商品营销内容生成

在嗅嗅豹内测过程中，我们发现选品分析虽然能帮助商家判断机会，但它在日常经营里相对低频。更高频的需求是：商家已经有一个正在售卖的商品，希望快速找到社媒里的同款爆款视频，理解它为什么能爆，并复刻成自己的商品营销视频。
